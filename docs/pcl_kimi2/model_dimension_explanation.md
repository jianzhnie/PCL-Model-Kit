# Kimi2-1T 模型参数维度解析

> 文件：`model_param_mapping_extracted.json` (通过 inspect dump 的 TP=1, PP=1, EP=1 配置，4层模型)
>
> 预训练脚本：`scripts/pretrain_kimi2_1t_4k.sh` (实际训练使用 TP=2, PP=8, EP=64)  
> 模型规模：1T 参数 (稀疏激活)，32 层

## ⚠️ 重要说明

本文档基于 `model_param_mapping_extracted.json`，这是通过 **inspect 方式 dump** 的模型参数，使用配置：

- **TP=1** (Tensor Parallel=1，无张量并行)
- **PP=1** (Pipeline Parallel=1，无流水线并行)  
- **EP=1** (Expert Parallel=1，无专家并行，包含**全部128个专家**)
- **层数**: 4 层 (Layer 0-3，2层 Dense + 2层 MoE)

而实际预训练使用的配置（见 `pretrain_kimi2_1t_4k.sh`）：

- **TP=2** (Tensor Parallel=2)
- **PP=8** (Pipeline Parallel=8)
- **EP=64** (Expert Parallel=64，每 rank 存 2 个专家)
- **CP=1** (Context Parallel=1，可选 Ulysses 算法)
- **层数**: 32 层 (Layer 0-1 Dense + Layer 2-31 MoE)

### 关键设计决策

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          Kimi2-1T 关键设计决策                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. 渐进式 MoE                                                               │
│     ┌─────────────────────────────────────────────────────────────────┐    │
│     │  Layer 0-1: Dense MLP (Dense FFN, hidden=18432)                 │    │
│     │         ↓ 学习基础语法/语义                                      │    │
│     │  Layer 2-31: MoE (Routed Experts, hidden=12288)                 │    │
│     │              学习抽象/多样化知识                                  │    │
│     └─────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  2. MLA (Multi-head Latent Attention)                                        │
│     ┌─────────────────────────────────────────────────────────────────┐    │
│     │  标准 GQA: 缓存 K, V → [b, 2, s, 128]                           │    │
│     │         ↓                                                       │    │
│     │  MLA: 缓存 K_latent, V_latent → 更小的 latent 表示              │    │
│     │       + q_layernorm([128]) + k_layernorm([128])                 │    │
│     │       → 节省 32 倍 KV Cache，支持 1M+ 上下文                    │    │
│     └─────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  3. MoE 架构设计                                                             │
│     ┌─────────────────────────────────────────────────────────────────┐    │
│     │  128 个专家                                                      │    │
│     │  ├── 每个 token 激活 2 个专家 (1.56% 稀疏度)                     │    │
│     │  ├── 1 个 Shared Expert (始终激活，捕获通用知识)                 │    │
│     │  └── EP=64: 64 个 rank 分摊，每 rank 存 2 个专家                 │    │
│     │       → 显存节省 64 倍                                          │    │
│     └─────────────────────────────────────────────────────────────────┘    │
│                                                                              │
│  4. 并行策略组合                                                             │
│     ┌─────────────────────────────────────────────────────────────────┐    │
│     │  TP=1: 不分割张量，保持完整维度                                  │    │
│     │  PP=1: 不分割层                                                │    │
│     │  EP=64: 专家并行，分摊 128 个专家                               │    │
│     │  CP=1: 序列并行 (可选)                                          │    │
│     └─────────────────────────────────────────────────────────────────┘    │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

| 第0维 | 163840 | 完整词表大小 `vocab_size`（TP=1 无需分割） |
| 第1维 | 7168   | 隐藏层维度 `hidden_size`            |


> **对比 TP=2 版本**：`[81920, 7168]` = 词表大小 ÷ 2

---

### 3.2 输出层 (Output Layer / LM Head)

```json
"module.output_layer.weight": [163840, 7168]
```

#### 3.2.1 作用与维度推导

输出层将 Decoder 最后一层的 hidden states 映射到词表空间，生成每个 token 的预测 logits：

```bash
输入: hidden_states [batch_size, seq_len, hidden_size]
                    = [b, s, 7168]

权重: W [vocab_size, hidden_size]
      = [163840, 7168]

输出: logits [batch_size, seq_len, vocab_size]
             = [b, s, 163840]

计算: logits = hidden_states @ W^T
      [b,s,7168] @ [7168,163840] = [b,s,163840]
```


| 维度  | 值      | 说明                           |
| --- | ------ | ---------------------------- |
| 第0维 | 163840 | 词表大小 `vocab_size`（TP=1 保持完整） |
| 第1维 | 7168   | 隐藏层维度 `hidden_size`          |


#### 3.2.2 为什么是 `[163840, 7168]` 而不是 `[7168, 163840]`？

这是 **Megatron/MindSpeed-LLM 的权重布局约定**，与 PyTorch `nn.Linear` 一致：

```python
# PyTorch nn.Linear 定义
nn.Linear(in_features=7168, out_features=163840)
# 内部权重存储为: [out_features, in_features] = [163840, 7168]

# 前向计算
output = input @ weight.T
# [b,s,7168] @ [7168,163840] = [b,s,163840]
```

Megatron 的 `ColumnParallelLinear` 实现：

```python
class ColumnParallelLinear:
    def __init__(self, input_size, output_size):
        # 权重: [output_size, input_size]
        self.weight = Parameter(torch.empty(output_size, input_size))
    
    def forward(self, input):
        # F.linear 内部执行: input @ weight.T
        return F.linear(input, self.weight, self.bias)
```

#### 3.2.3 TP 对输出层的影响


| 配置   | 输出层维度            | 说明         |
| ---- | ---------------- | ---------- |
| TP=1 | `[163840, 7168]` | 完整词表       |
| TP=2 | `[81920, 7168]`  | 词表按列分割 ÷ 2 |
| TP=4 | `[40960, 7168]`  | 词表按列分割 ÷ 4 |


TP>1 时，输出层采用 **Column Parallel**，每个 rank 负责部分词汇的预测，最后通过 `gather` 合并完整 logits。

#### 3.2.4 与 Embedding 的权重关系

预训练脚本配置：

```bash
--untie-embeddings-and-output-weights  # 不共享权重
```

这意味着：

```json
"module.embedding.word_embeddings.weight": [163840, 7168]   # 独立
"module.output_layer.weight": [163840, 7168]                # 独立
```

**参数对比**：

```
共享权重时:  163840 × 7168 ≈ 1.17B 参数
不共享时:    163840 × 7168 × 2 ≈ 2.35B 参数
额外开销:    +1.17B 参数
```

大多数现代 LLM（如 GPT-3、Llama）选择**共享权重**，因为：

1. 减少约 1B+ 参数
2. 嵌入层和输出层语义相关（都是词向量空间）
3. 训练更稳定

但 Kimi2 选择不共享，可能原因：

- 解耦输入/输出表示，增加模型容量
- 支持不同的初始化策略
- 便于后续微调或蒸馏

### 3.3 Attention 层

#### 3.3.0 Attention 机制演进：从 MHA 到 GQA 再到 MLA

在深入 Kimi2 的 Attention 实现之前，让我们先理解现代大模型 Attention 机制的演进路径。

---

**三种 Attention 架构对比**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Attention 机制演进路线                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   标准 MHA (Multi-Head Attention)                                           │
│   ├── Q: 64 heads × 128 = 8192 dim                                          │
│   ├── K: 64 heads × 128 = 8192 dim  ◄── 每个头独立的 K/V                   │
│   └── V: 64 heads × 128 = 8192 dim                                          │
│       │                                                                      │
│       ▼ 问题：KV Cache 太大，推理内存爆炸                                    │
│                                                                              │
│   GQA (Grouped Query Attention)                                             │
│   ├── Q: 64 heads × 128 = 8192 dim                                          │
│   ├── K: 2 groups × 128 = 256 dim   ◄── KV 分组共享 (64→2)                 │
│   └── V: 2 groups × 128 = 256 dim                                           │
│       │                                                                      │
│       ▼ 改进：KV Cache 减少 32 倍，但仍缓存完整维度                          │
│                                                                              │
│   MLA (Multi-head Latent Attention) ⭐ Kimi2 使用                            │
│   ├── Q: 64 heads × 128 = 8192 dim                                          │
│   ├── K_latent: 2 × 128 = 256 dim   ◄── 压缩到 latent space                │
│   └── V_latent: 2 × 128 = 256 dim   ◄── 通过 q/k_layernorm 进一步压缩      │
│       │                                                                      │
│       ▼ 突破：缓存压缩表示，可扩展至 32×+ 内存节省                            │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

---

##### 1. 标准 MHA（基础版）

```python
# 标准 Multi-Head Attention
Q = X @ W_q  # [batch, seq, 8192]  (64 heads × 128)
K = X @ W_k  # [batch, seq, 8192]  (64 heads × 128)
V = X @ W_v  # [batch, seq, 8192]  (64 heads × 128)

# KV Cache 大小：每层需要缓存 8192 dim 的 K 和 V
# 128K 上下文 × 8192 × 2(K+V) × 32层 × 2字节(fp16) = 32.8 GB！
```

**问题**：每个注意力头都有独立的 K/V，导致 KV Cache 随层数和头数线性增长。

---

##### 2. GQA（Grouped Query Attention）

GQA 是 MHA 和 MQA（Multi-Query Attention）的折中方案：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        GQA 架构可视化                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   输入 X: [batch, seq_len, 7168]                                            │
│       │                                                                      │
│       ├───► Q 投影 ───► [batch, seq, 8192] ◄── 64 个 Query 头               │
│       │                    │                                                 │
│       │                    ▼                                                 │
│       │              ┌─────────────┐                                         │
│       │              │ 64 heads    │                                         │
│       │              │ [128 dim]   │                                         │
│       │              └──────┬──────┘                                         │
│       │                     │                                                │
│       ├───► K 投影 ───► [batch, seq, 256] ◄── 2 个 KV 组 (GQA)              │
│       │                    │                                                 │
│       │                    ▼                                                 │
│       │              ┌─────────────┐                                         │
│       │              │ 2 groups    │ ◄── 每 32 个 Q heads 共享 1 组 KV      │
│       │              │ [128 dim]   │                                         │
│       │              └──────┬──────┘                                         │
│       │                     │                                                │
│       └───► V 投影 ───► [batch, seq, 256] ◄── 2 个 KV 组 (GQA)              │
│                            │                                                 │
│                            ▼                                                 │
│                      ┌─────────────┐                                         │
│                      │ 2 groups    │                                         │
│                      │ [128 dim]   │                                         │
│                      └─────────────┘                                         │
│                                                                              │
│   KV Cache: 只需缓存 256 dim (vs MHA 的 8192 dim)                           │
│   内存节省: 8192 / 256 = 32 倍                                              │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

**GQA 核心思想**：
- Query 保持 64 个头（保证表达能力）
- K/V 分成 2 个组，每组被 32 个 Query 头共享
- KV Cache 从 8192 dim 降到 256 dim

**GQA 的优势**：
1. 显著减少 KV Cache 内存（32 倍）
2. 比 MQA（单组 KV）表达能力强
3. 实现简单，仅需修改 KV 投影维度

**GQA 的局限**：
- 仍缓存完整的 K/V 维度（256 dim）
- 没有进一步压缩 latent 表示

---

##### 3. MLA（Multi-head Latent Attention）⭐

MLA 是 DeepSeek-V2/V3 和 Kimi2 采用的技术，在 GQA 基础上进一步压缩：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        MLA 架构可视化                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   输入 X: [batch, seq_len, 7168]                                            │
│       │                                                                      │
│       ├───► Q 投影 ─────────────────────────────┐                           │
│       │              [batch, seq, 8192]         │                           │
│       │                    │                    │                           │
│       │                    ▼                    │                           │
│       │           ┌───────────────┐            │                           │
│       │           │ q_layernorm   │ ◄── MLA 特有！                         │
│       │           │ [128]         │            │                           │
│       │           └───────┬───────┘            │                           │
│       │                   │                     │                           │
│       │                   ▼                     │                           │
│       │           Q_latent: [b,s,8192]         │                           │
│       │           (压缩后的 Query 表示)         │                           │
│       │                                         │                           │
│       ├───► K 投影 ───────► [batch, seq, 256]  │                           │
│       │                           │             │                           │
│       │                           ▼             │                           │
│       │                    ┌───────────┐       │                           │
│       │                    │k_layernorm│ ◄── MLA 特有！                    │
│       │                    │  [128]    │       │                           │
│       │                    └─────┬─────┘       │                           │
│       │                          │              │                           │
│       │                          ▼              │                           │
│       │                   K_latent: [b,s,256]  │                           │
│       │                   (缓存这个!)           │                           │
│       │                                         │                           │
│       └───► V 投影 ───────► [batch, seq, 256] ─┘                           │
│                                   │                                         │
│                                   ▼                                         │
│                            V_latent: [b,s,256]                             │
│                            (缓存这个!)                                      │
│                                                                              │
│   ┌─────────────────────────────────────────────────────────────────────┐  │
│   │                      KV Cache 对比                                  │  │
│   ├─────────────────────────────────────────────────────────────────────┤  │
│   │                                                                     │  │
│   │   GQA:  缓存 K [b,s,256], V [b,s,256]                              │  │
│   │        = 512 dim 每层                                              │  │
│   │                                                                     │  │
│   │   MLA:  缓存 K_latent [b,s,256], V_latent [b,s,256]                │  │
│   │        = 512 dim 每层                                              │  │
│   │        + 通过压缩/解压缩可进一步扩展至更小 latent                   │  │
│   │                                                                     │  │
│   │   关键：MLA 的 256 dim 是压缩表示，可通过 W_k_up/W_v_up 解压缩      │  │
│   │                                                                     │  │
│   └─────────────────────────────────────────────────────────────────────┘  │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

**MLA 核心创新**：

```python
# 1. 低秩压缩投影
Q = X @ W_q              # [b, s, 8192]
K_latent = X @ W_k       # [b, s, 256]  (压缩)
V_latent = X @ W_v       # [b, s, 256]  (压缩)

# 2. MLA 特有的 LayerNorm 稳定训练
K_latent = k_layernorm(K_latent)  # [b, s, 256]
V_latent = v_layernorm(V_latent)  # [b, s, 256]

# 3. 解压缩到完整维度 (推理时)
K = K_latent @ W_k_up    # [b, s, 256] @ [256, 8192] = [b, s, 8192]
V = V_latent @ W_v_up    # [b, s, 256] @ [256, 8192] = [b, s, 8192]

# 4. Attention 计算 (与 GQA 相同)
attn_output = softmax(Q @ K^T / √d) @ V
```

|------|-----------|-----|-------------|

| **KV 头数/组数** | 64 | 2 groups | 2 groups |
| **每头维度** | 128 | 128 | 128 |

| **K/V 维度** | 8192 | 256 | 256 |
| **KV Cache (每层)** | 16384 dim | 512 dim | 512 dim |

| **k_layernorm** | ❌ 无 | ❌ 无 | ✅ 有 [128] |
| **Latent 压缩** | ❌ 无 | ❌ 无 | ✅ 有 |
| **解压缩矩阵** | 不需要 | 不需要 | W_k_up, W_v_up |
| **内存节省** | 1× (基准) | 32× | 32×+ (可扩展) |
| **表达能力** | 最强 | 中等 | 强 (通过解压缩恢复) |
| **训练稳定性** | 高 | 高 | 中等 (需 layernorm) |

##### 6. Kimi2 MLA 配置速查

```yaml
# Kimi2-1T MLA 配置
num_attention_heads: 64        # 64 个 Query 头
num_query_groups: 2            # 2 个 KV 组 (GQA 特征)
kv_channels: 128               # 每个头 128 维

# MLA 特有参数
q_layernorm: [128]             # Query 压缩后的 LayerNorm
k_layernorm: [128]             # Key 压缩后的 LayerNorm

# 维度计算
Q_dim: 64 × 128 = 8192        # 64 heads × 128 dim
K_dim: 2 × 128 = 256          # 2 groups × 128 dim
V_dim: 2 × 128 = 256          # 2 groups × 128 dim

# 合并 QKV 投影
linear_qkv: [8704, 7168]       # 8192 + 256 + 256 = 8704
```

---

#### 3.3.1 QKV 投影层 (linear_qkv)

```json
"module.decoder.layers.0.self_attention.linear_qkv.weight": [8704, 7168]
```

##### 维度计算原理

`linear_qkv` 将 Q、K、V 三个投影**合并**到一个线性层中，这是 Megatron 和 MLA 架构的常见优化：

```python
# 关键配置参数
num_attention_heads = 64      # 注意力头数
num_query_groups = 2          # GQA 的 KV 组数 (GQA: Grouped Query Attention)
kv_channels = 128             # 每个头的维度
hidden_size = 7168            # 输入隐藏层维度

# QKV 输出维度详细计算
Q_dim = num_attention_heads × kv_channels = 64 × 128 = 8192
K_dim = num_query_groups × kv_channels = 2 × 128 = 256
V_dim = num_query_groups × kv_channels = 2 × 128 = 256

# 合并后的总输出维度
total_dim = Q_dim + K_dim + V_dim = 8192 + 256 + 256 = 8704
```


| 维度  | 值    | 计算方式                           |
| --- | ---- | ------------------------------ |
| 第0维 | 8704 | Q(8192) + K(256) + V(256) 合并输出 |
| 第1维 | 7168 | 输入维度 `hidden_size`             |


##### 为什么是 MLA 架构？

从参数文件中可以确认这是 **MLA (Multi-head Latent Attention)**，而非标准 GQA：

```json
"module.decoder.layers.0.self_attention.q_layernorm.weight": [128]
"module.decoder.layers.0.self_attention.k_layernorm.weight": [128]
```

**标准 GQA** 没有这两个参数，MLA 特有！


| 特性对比          | 标准 GQA        | MLA (当前)                               |
| ------------- | ------------- | -------------------------------------- |
| Q 维度          | 8192 (64×128) | 8192 (64×128)                          |
| K 维度          | 256 (2×128)   | 256 (2×128)                            |
| V 维度          | 256 (2×128)   | 256 (2×128)                            |
| **Q/K 压缩**    | 无             | **有** (通过 `q_layernorm`/`k_layernorm`) |
| **Latent 空间** | 等于 head_dim   | 等于 `kv_channels=128`                   |
| **KV Cache**  | 缓存完整 K/V      | 缓存压缩后的 latent 表示                       |


##### 计算流程详解

```python
# 1. 输入 hidden_states
hidden = input(...)  # [batch, seq_len, 7168]

# 2. 通过 linear_qkv 合并投影 (一次矩阵乘法完成 QKV)
qkv = hidden @ linear_qkv.weight.T  # [b, s, 8704]

# 3. 切分成 Q、K、V
q = qkv[..., 0:8192]           # [b, s, 8192]  ← 64 个头，每个 128 维
k = qkv[..., 8192:8448]        # [b, s, 256]   ← 2 个 KV 组，每个 128 维
v = qkv[..., 8448:8704]        # [b, s, 256]   ← 2 个 KV 组，每个 128 维

# 4. MLA 特有的压缩和标准化
# 将 Q/K 从原始维度压缩到 latent space
q_compressed = q_layernorm(q)   # [b, s, 128] 或保持 [b, s, 8192] 但做 norm
k_compressed = k_layernorm(k)   # [b, s, 128] 或保持 [b, s, 256] 但做 norm

# 5. Attention 计算
# Q: [b, s, 8192] 分成 64 个头 → [b, 64, s, 128]
# K: [b, s, 256] 分成 2 个组  → [b, 2, s, 128] (广播到 64 个头)
# V: [b, s, 256] 分成 2 个组  → [b, 2, s, 128] (广播到 64 个头)
attention_output = attention(q, k, v)
```

##### Sliding Window Attention (SWA)

```bash
--swa-windows 128           # SWA 窗口大小
--full-attention-layers     # 指定哪些层使用完整 attention
--mla-fa-divide-qk          # MLA FlashAttention QK 分离
```

**SWA 设计**：

```
标准 Attention:
每个 token 可以看到所有前面的 token
计算复杂度: O(n²)

Sliding Window Attention:
每个 token 只能看到窗口内的 token（如 128）
计算复杂度: O(n × window_size)

适用于:
- 超长上下文（128K）
- 局部模式更明显的场景

Kimi2 实现:
- 大多数层使用 SWA (window=128)
- 部分层（如第 0, 16, 31 层）使用完整 attention
- 捕获局部模式 + 全局信息
```

```bash
--rope-scaling-original-max-position-embeddings 4096
```


##### TP 并行对比


| 配置   | linear_qkv 维度  | QKV 输出被切分方式              |

| TP=1 | `[8704, 7168]` | 完整保持 8704                |

TP (张量并行) 将 QKV 的输出维度**按列分割**，每个 rank 负责计算部分 Q、K、V。

##### 为什么合并 QKV？

```
优势:
1. 减少内存访问次数 (1 次大的矩阵乘法 vs 3 次小的)
2. 更好地利用 GPU 计算单元 (更大的 GEMM 更高效)
3. 与 Megatron 的并行策略对齐

劣势:
1. 灵活性降低 (无法单独对 Q/K/V 使用不同的并行策略)
2. 但对于标准 Transformer，QKV 通常一起计算，所以影响不大
```

#### 3.3.2 Output 投影层 (linear_proj)

```json
"module.decoder.layers.0.self_attention.linear_proj.weight": [7168, 8192]
```

##### 作用与维度推导

`linear_proj` 是 Attention 机制的**输出投影层**（也称为 O-Projection 或 Output Projection），将多头注意力的结果映射回 `hidden_size`：

```python
# 关键配置参数
num_attention_heads = 64      # 注意力头数
kv_channels = 128             # 每个头的维度
hidden_size = 7168            # 模型隐藏层维度

# 输入: Attention 计算后的拼接结果
# 来自 64 个头的输出，每个头 128 维
attention_output_dim = num_attention_heads × kv_channels = 64 × 128 = 8192

# 输出投影将多头结果映射回 hidden_size
output_dim = hidden_size = 7168

# 权重形状: [output_dim, input_dim] = [7168, 8192]
```


| 维度  | 值    | 说明                                                |
| --- | ---- | ------------------------------------------------- |
| 第0维 | 7168 | 输出维度 `hidden_size`，与输入 hidden 对齐                  |
| 第1维 | 8192 | 输入维度，来自 64 个头 × 128 维 = `num_heads × kv_channels` |


##### 在 Attention 流程中的位置

```
完整 Attention 流程:

输入 hidden: [b, s, 7168]
       │
       ▼
┌─────────────────────────────────────┐
│  linear_qkv: [8704, 7168]           │
│  输出 QKV: [b, s, 8704]              │
│  (8192 Q + 256 K + 256 V)            │
└─────────────────────────────────────┘
       │
       ▼ 切分 Q, K, V
┌─────────────────────────────────────┐
│  Attention 计算:                     │
│  Q: [b, 64, s, 128] (64 个头)        │
│  K: [b, 2, s, 128]  (2 个 KV 组)     │
│  V: [b, 2, s, 128]  (2 个 KV 组)     │
│                                     │
│  softmax(Q @ K^T / √d) @ V          │
│                                     │
│  输出: [b, 64, s, 128]              │
└─────────────────────────────────────┘
       │
       ▼ reshape + concat
┌─────────────────────────────────────┐
│  多头拼接结果: [b, s, 8192]          │
│  (64 个头 × 128 维 = 8192)           │
└─────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────┐

│  输出: [b, s, 7168]                  │
└─────────────────────────────────────┘
       │
       ▼
残差连接: output + input
       │
       ▼
LayerNorm
```

##### 计算过程详解

```python
# 1. Attention 计算后的多头输出
# 来自 64 个注意力头，每个头输出 128 维
multi_head_output = attention(q, k, v)  # [b, 64, s, 128]

# 2. reshape: 将多头结果拼接
# [b, 64, s, 128] -> [b, s, 64×128] = [b, s, 8192]
concatenated = multi_head_output.transpose(1, 2).reshape(b, s, 8192)

# 3. 输出投影 (linear_proj)
# weight: [7168, 8192]
# 计算: output = concatenated @ weight.T
output = linear_proj(concatenated)  # [b, s, 8192] @ [8192, 7168] = [b, s, 7168]

# 4. 残差连接 (Transformer 标准结构)
final_output = output + residual  # [b, s, 7168] + [b, s, 7168]
```

##### 为什么是 [7168, 8192] 而不是 [8192, 7168]？

再次确认 Megatron 的权重布局：

```python
# Megatron/PyTorch 约定: weight 形状为 [out_features, in_features]
linear_proj = ColumnParallelLinear(
    in_features=8192,   # 多头拼接后的维度
    out_features=7168   # 投影回 hidden_size
)
# 内部存储: weight = [out, in] = [7168, 8192]

# 前向计算:
# output = input @ weight.T
# [b, s, 8192] @ [8192, 7168] = [b, s, 7168]
```

##### TP 并行影响


| 配置   | linear_proj 维度 | 切分方式                 |
| ---- | -------------- | -------------------- |
| TP=1 | `[7168, 8192]` | 完整权重                 |
| TP=2 | `[7168, 4096]` | 输入维度 8192 ÷ 2 = 4096 |
| TP=4 | `[7168, 2048]` | 输入维度 8192 ÷ 4 = 2048 |


**注意**：`linear_proj` 使用 **RowParallelLinear**（行并行），按输入维度切分：

- 每个 rank 负责计算部分头的输出
- 最后通过 `reduce_scatter` 或 `all_reduce` 合并结果
- 输出维度 7168 保持不变（所有 rank 都有完整的输出维度）

#### 3.3.3 MLA 特有的 LayerNorm

```json
"module.decoder.layers.0.self_attention.q_layernorm.weight": [128]
"module.decoder.layers.0.self_attention.k_layernorm.weight": [128]
```

##### 什么是 MLA?

**MLA (Multi-head Latent Attention)** 是 DeepSeek-V2/V3 和 Kimi2 采用的一种注意力机制优化，核心思想是：

> 将 Key 和 Value 压缩到更小的 **latent space**，大幅减少 KV Cache 内存占用。

##### 维度解释


| 参数                   | 维度    | 含义                             |
| -------------------- | ----- | ------------------------------ |
| `q_layernorm.weight` | [128] | 对 Query 在 latent 空间做 LayerNorm |
| `k_layernorm.weight` | [128] | 对 Key 在 latent 空间做 LayerNorm   |


**128** = `kv_channels`（每个注意力头的通道数）

##### 为什么 MLA 需要这两个 LayerNorm？

在标准 Multi-Head Attention 中：

```python
# 标准 MHA
Q = X @ W_q    # [b, s, 8192]
K = X @ W_k    # [b, s, 256]
V = X @ W_v    # [b, s, 256]

# 直接计算 attention
attn = softmax(Q @ K^T) @ V
```

在 **MLA** 中，采用**低秩压缩**策略：

```python
# MLA: 压缩 K/V 到 latent space
Q = X @ W_q              # [b, s, 8192]
K_latent = X @ W_k       # [b, s, 256]  (压缩后)
V_latent = X @ W_v       # [b, s, 256]  (压缩后)

# 在 latent space 做 LayerNorm，稳定训练
K_latent = k_layernorm(K_latent)   # [b, s, 256] -> 在 128 维上 norm
V_latent = v_layernorm(V_latent)   # [b, s, 256]

# 解压缩并计算 attention
K = K_latent @ W_k_up    # [b, s, 256] @ [256, 8192] = [b, s, 8192]
V = V_latent @ W_v_up    # [b, s, 256] @ [256, 8192] = [b, s, 8192]

attn = softmax(Q @ K^T) @ V
```

**关键洞察**：

- 缓存时只存 **K_latent/V_latent** (256 维)，而不是完整的 K/V (8192 维)
- **节省 32 倍 KV Cache 内存**！
- `q_layernorm` 和 `k_layernorm` 用于**稳定压缩/解压缩过程**

##### 维度为什么是 [128]？

```
kv_channels = 128  (每个头的维度)

k_layernorm.weight 的形状分析:
- 输入 K_latent: [b, s, 256] = [b, s, 2 groups × 128]
- LayerNorm 作用在特征维度上
- 但 MLA 中 Q/K 会经过压缩，在 128 维的 latent 空间做 normalization

实际上，在 Megatron 的 MLA 实现中:
q_layernorm:  对 query 的每个 128 维头做 norm
k_layernorm:  对 key 的每个 128 维 latent 做 norm

所以 weight 形状是 [128]，对应每个头的通道数
```

| `k_layernorm` | 对 Key latent 做标准化 | K 压缩后，解压缩前          |


**为什么需要它们？**

1. **稳定训练**：压缩/解压缩过程容易数值不稳定，LayerNorm 提供稳定性
2. **标准化表示**：让不同 layer 的 latent 表示在同一尺度
3. **MLA 特有**：标准 Attention 不需要，因为直接计算没有压缩/解压缩

| **k_layernorm** | ❌ 无          | ✅ 有 [128]           |
| **解压缩**         | ❌ 不需要        | ✅ W_k_up, W_v_up    |
| **内存节省**        | 1× (GQA 基准)  | 可扩展至 32×+ (大规模 MLA) |


##### 总结

```
q_layernorm.weight [128] 和 k_layernorm.weight [128] 是 MLA 架构的"指纹":

1. 它们的存在证明了这个模型使用 MLA，而非标准 Attention
2. 128 = kv_channels，对应每个注意力头的通道数
3. 它们在 latent space 做标准化，稳定压缩/解压缩过程
4. 使得模型可以缓存更小的 K/V latent，大幅节省推理内存

这是 Kimi2 能够支持超长上下文 (1M+) 的关键技术之一！
```

```python
# 从预训练脚本中提取的关键配置
hidden_size = 7168          # 模型隐藏层维度
ffn_hidden_size = 18432     # Dense FFN 层的中间维度
```

**第二步：理解 SwiGLU 的结构**

SwiGLU 激活函数需要**两个独立的线性投影**：

```
┌─────────────────────────────────────────────────────────────┐
│                    SwiGLU 结构详解                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   输入: [b, s, 7168]                                        │
│        │                                                     │
│        ├───► Gate 投影 ───► [b, s, 18432]  ──┐              │
│        │                    (控制信息流)      │              │
│        │                                     ├──► 元素乘法   │
│        │                    (特征上采样)      │   (gate * up) │
│        └───► Up 投影   ───► [b, s, 18432]  ──┘              │
│        │                                                     │
│        │                                                    │
│   合并这两个投影的权重:                                       │
│   Gate: [18432, 7168]                                       │
│   Up:   [18432, 7168]                                       │
│                                                              │
│   总输出维度 = 18432 + 18432 = 36864                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**第三步：数学计算**

```
linear_fc1 第0维 = ffn_hidden_size (gate) + ffn_hidden_size (up)
                 = 18432 + 18432
                 = 36864

或者等价地:
linear_fc1 第0维 = ffn_hidden_size × 2
                 = 18432 × 2
                 = 36864
```

**第四步：验证维度**

```python
# 权重矩阵形状验证
linear_fc1.weight.shape = [36864, 7168]
                        = [输出维度, 输入维度]
                        = [gate(18432) + up(18432), hidden_size]

# 切分后
gate_weight = weight[0:18432, :]      # [18432, 7168]
up_weight   = weight[18432:36864, :]  # [18432, 7168]
```

**直观理解**：

```
linear_fc1 的 36864 行可以分为两部分:
┌─────────────────────────────────────────────────────────┐
│  第 0 ~ 18431 行:  Gate 投影权重  [18432, 7168]          │
├─────────────────────────────────────────────────────────┤
│  第 18432 ~ 36863 行: Up 投影权重  [18432, 7168]         │
└─────────────────────────────────────────────────────────┘
                    总计 36864 行
```

**对比其他激活函数**：


| 激活函数       | FFN 结构           | linear_fc1 维度   | 计算公式                  |
| ---------- | ---------------- | --------------- | --------------------- |
| **SwiGLU** | Gate + Up + Down | `[36864, 7168]` | `2 × ffn_hidden_size` |
| GELU       | Up + Down        | `[18432, 7168]` | `1 × ffn_hidden_size` |
| ReLU       | Up + Down        | `[18432, 7168]` | `1 × ffn_hidden_size` |


SwiGLU 比传统激活函数多一个 Gate 投影，所以维度翻倍！

##### SwiGLU 激活函数

**SwiGLU** 是目前大模型（Llama、Qwen、DeepSeek 等）最常用的 FFN 激活函数：

```python
# SwiGLU 公式
SwiGLU(x, W_gate, W_up, W_down) = (silu(x @ W_gate) * (x @ W_up)) @ W_down

其中:
- silu(x) = x * sigmoid(x)  (也称为 Swish)
- W_gate: gate 投影权重
- W_up: up 投影权重  
- W_down: down 投影权重
```

**在 Megatron 中的实现**：

```python
# linear_fc1 合并了 gate 和 up 两个投影
# weight 形状: [36864, 7168] = [2 * ffn_hidden_size, hidden_size]

# 实际使用时切分
fc1_output = hidden @ linear_fc1.weight.T  # [b, s, 36864]
gate, up = fc1_output.chunk(2, dim=-1)     # 各 [b, s, 18432]

# SwiGLU 计算
activated = F.silu(gate) * up              # [b, s, 18432]

# Down 投影
output = activated @ linear_fc2.weight.T   # [b, s, 7168]
```

##### 维度变化流程

```
输入 hidden: [b, s, 7168]
       │
       ▼
┌───────────────────────────────────────────┐
│  linear_fc1: [36864, 7168]                │
│  输出: [b, s, 36864]                       │
│  = [b, s, gate(18432) + up(18432)]        │
└───────────────────────────────────────────┘
       │
       ▼ 切分成 gate 和 up
┌───────────────────────────────────────────┐
│  gate: [b, s, 18432]                       │
│  up:   [b, s, 18432]                       │
│                                            │
│  SwiGLU: silu(gate) * up                   │
│  输出: [b, s, 18432]                       │
└───────────────────────────────────────────┘
       │
       ▼

```

##### 为什么 Gate + Up 要合并？

```
优势:
1. 单次矩阵乘法完成两个投影，减少内存访问
2. 更大的 GEMM 更充分利用 GPU 计算能力
3. Megatron 的并行策略可以统一处理

合并后: [hidden_size, 2 * ffn_hidden_size]
      = [7168, 36864]

存储布局 (按行):
┌─────────────────────────────────────────────────────┐
│  W_gate: [7168, 18432]  前一半                      │
│  W_up:   [7168, 18432]  后一半                      │
└─────────────────────────────────────────────────────┘
```

### 3.5 MoE 层（第 2 层开始）

从第 2 层开始，模型使用 **Mixture of Experts (MoE)** 替代 Dense MLP。这是 Kimi2-1T 模型的核心设计，使得 1T 参数规模成为可能。

#### 3.5.0 MoE 架构概览

**为什么使用 MoE？**

```
传统 Dense 模型:
- 每个 token 经过相同的 FFN
- 参数量 = 计算量
- 1T 参数意味着每次前向传播都要计算 1T 参数

MoE 模型:
- 每个 token 只激活部分专家（这里是 2/128）
- 参数量 ≠ 计算量
- 1T 参数，但每次只计算 ~16B 参数（稀疏激活）
```

**Kimi2 的 MoE 配置**（从预训练脚本提取）：

```bash
--num-experts 128           # 总专家数
--moe-router-topk 2         # 每个 token 选择 2 个专家
--n-shared-experts 1        # 1 个共享专家
--moe-ffn-hidden-size 12288 # 专家 FFN 维度
--expert-model-parallel-size 64  # EP=64
--moe-router-score-function sigmoid
--moe-router-enable-expert-bias  # 启用专家偏置
```

**MoE 层的三个核心组件**：

```
输入 hidden: [b, s, 7168]
       │
       ├───► Router ───► 选择 Top-2 专家
       │
       ├───► Shared Expert ───► 始终激活
       │
       └───► Routed Experts ───► 只激活选中的 2 个
       │
       ▼
  加权合并三个输出
       │
       ▼
  输出: [b, s, 7168]
```

#### 3.5.1 Router (门控网络)

```json
"module.decoder.layers.2.mlp.router.weight": [128, 7168]
```

##### 3.5.1.1 Router 的作用

Router 是 MoE 的"交通指挥官"，决定每个 token 应该由哪些专家处理：

```python
# Router 前向计算流程
hidden = input(...)  # [batch, seq_len, 7168]

# 计算每个 token 对每个专家的分数
logits = hidden @ router.weight.T  # [b, s, 7168] @ [7168, 128] = [b, s, 128]

# 应用 softmax 得到概率分布
scores = sigmoid(logits)  # [b, s, 128]，使用 sigmoid 而非 softmax

# 选择 Top-K 专家（K=2）
topk_scores, topk_indices = torch.topk(scores, k=2, dim=-1)
# topk_scores:  [b, s, 2]  （选中的专家的分数）
# topk_indices: [b, s, 2]  （选中的专家的索引，范围 0-127）

# 归一化 Top-K 分数
topk_scores = topk_scores / topk_scores.sum(dim=-1, keepdim=True)
```

##### 3.5.1.2 维度解析


| 维度  | 值    | 说明                           |
| --- | ---- | ---------------------------- |
| 第0维 | 128  | `num_experts`，完整专家数，每个专家一个分数 |
| 第1维 | 7168 | 输入维度 `hidden_size`           |


**为什么是 [128, 7168]？**

```
Router 是一个简单的线性层:
- 输入: [b, s, 7168] (token 的 hidden representation)
- 权重: [128, 7168]
- 计算: [b, s, 7168] @ [7168, 128] = [b, s, 128]
- 输出: 每个 token 对 128 个专家的分数

这相当于有 128 个"专家向量"，每个 7168 维
计算输入与每个专家向量的相似度
```

##### 3.5.1.3 负载均衡（Load Balancing）

MoE 训练面临一个核心问题：**负载均衡**。如果 Router 总是选择相同的几个专家，其他专家就无法被训练。

**Kimi2 使用的负载均衡策略**：

```bash
--moe-router-load-balancing-type aux_loss  # 使用辅助损失
--moe-aux-loss-coeff 0.01                  # 辅助损失系数
--moe-z-loss-coeff 0.001                   # Z-loss 系数
--seq-aux                                   # 序列级辅助损失
--norm-topk-prob                            # 归一化 Top-K 概率
```

**辅助损失计算**：

```python
# 目标：让每个专家接收大致相同数量的 token

# 1. 计算每个专家接收的 token 比例
f_i = (选中了专家 i 的 token 数) / (总 token 数)  # 实际负载

# 2. 计算每个专家的路由概率平均
P_i = mean(所有 token 对专家 i 的分数)  # 期望负载

# 3. 辅助损失 = num_experts × Σ(f_i × P_i)
aux_loss = 128 × sum(f_i * P_i for i in range(128))

# 最小化 aux_loss → f_i 和 P_i 都趋向于 1/128
# 即每个专家处理 1/128 的 token
```

**Z-loss**：防止 Router 输出极端值（某些分数接近 1，其他接近 0）

```python
# Z-loss 鼓励 Router 输出更"平缓"的分布
z_loss = log(sum(exp(logits))) ** 2
```

##### 3.5.1.4 Group-Limited Routing (组限制路由)

```bash
--moe-router-num-groups 8       # 将 128 个专家分成 8 组
--moe-router-group-topk 2       # 从每组中选择 top-2
--moe-router-topk 2             # 最终选择 2 个专家
```

**Group-Limited Routing 设计**：

```
标准 MoE 路由:
- 128 个专家中选择 top-2
- 通信复杂度: O(128) 需要与所有专家所在的 rank 通信

Group-Limited Routing:
- 128 个专家分成 8 组，每组 16 个专家
- 先选择 top-2 组? (实际实现略有不同)
- 从选择的组中选择 top-2 专家
- 限制通信范围，提高效率
```

**实际计算流程**：

```python
# 1. 将 128 个专家分成 8 组，每组 16 个专家
num_groups = 8
experts_per_group = 128 // 8 = 16

# 2. 计算每组的聚合分数（或选择策略）
group_scores = ...  # [b, s, 8]

# 3. 选择 top 组 (实际实现可能有变化)
# 4. 在选中的组内选择 top-2 专家

# 最终: topk_indices [b, s, 2]
# 选中的专家索引范围 0-127
```

> 注意：具体实现可能因 MindSpeed-LLM 版本而异，但核心思想是限制路由选择的范围以减少通信。

#### 3.5.2 Shared Expert (共享专家)

```json
"module.decoder.layers.2.mlp.shared_experts.linear_fc1.weight": [24576, 7168]

```

##### 3.5.2.1 为什么需要 Shared Expert？

**问题**：纯 MoE 设计可能导致通用知识分散在各个专家中

**解决方案**：引入 Shared Expert 捕获通用知识

```
Shared Expert 特点:
1. 所有 token 都经过 Shared Expert（100% 激活率）
2. 与 Routed Experts 并行计算
3. 输出加权合并
4. 减少 Routed Experts 学习通用知识的压力
```

##### 3.5.2.2 维度解析

**linear_fc1 (Gate + Up)**:


| 维度  | 值     | 计算方式                                          |
| --- | ----- | --------------------------------------------- |
| 第0维 | 24576 | `moe_ffn_hidden_size × 2 = 12288 × 2`（SwiGLU） |
| 第1维 | 7168  | 输入维度 `hidden_size`                            |


**与 Dense MLP 的对比**：

- Dense MLP: `36864 = 18432 × 2`（ffn_hidden_size=18432）
- Shared Expert: `24576 = 12288 × 2`（moe_ffn_hidden_size=12288）

Shared Expert 的 FFN 维度更小，因为 Routed Experts 也参与计算。

**linear_fc2 (Down)**:


| 维度  | 值     | 说明                    |
| --- | ----- | --------------------- |
| 第0维 | 7168  | 输出维度 `hidden_size`    |
| 第1维 | 12288 | `moe_ffn_hidden_size` |


##### 3.5.2.3 Shared Expert 计算流程

```python
# Shared Expert 对所有 token 计算
hidden = input(...)  # [b, s, 7168]

# Gate + Up 投影 (SwiGLU)
shared_fc1_out = hidden @ shared_fc1.weight.T  # [b, s, 24576]
gate, up = shared_fc1_out.chunk(2, dim=-1)     # 各 [b, s, 12288]
shared_activated = F.silu(gate) * up           # [b, s, 12288]

# Down 投影
shared_output = shared_activated @ shared_fc2.weight.T  # [b, s, 7168]

# 最终输出会与 Routed Experts 的输出加权合并
```

#### 3.5.3 Routed Experts (路由专家) ⭐ 核心设计

```json
"module.decoder.layers.2.mlp.experts.weight1": [7168, 3145728]   // 128个专家 Gate+Up
"module.decoder.layers.2.mlp.experts.weight2": [1572864, 7168]   // 128个专家 Down
```

**重要说明**：由于 dump 配置使用 EP=1（无专家并行），权重包含了**全部128个专家**。

- 实际训练 EP=64 时，每个 rank 只存储 2 个专家：
  - `weight1`: [7168, 49152] = [7168, 24576 × 2]
  - `weight2`: [24576, 7168] = [12288 × 2, 7168]

##### 3.5.3.1 EP (Expert Parallel) 并行策略

**关键配置**：

- 总专家数: 128
- EP (Expert Parallel): 64
- 每个 EP rank 负责的专家数: `128 ÷ 64 = 2` 个专家

**EP 并行的核心思想**：

```
不使用 EP (单卡):
- 1 张卡存储 128 个专家的所有权重
- 显存需求巨大: 128 × (专家参数量)

使用 EP=64:
- 64 张卡分摊 128 个专家
- 每张卡只存储 2 个专家的权重
- 显存需求降为 1/64！

通信开销:
- Router 决定 token 分配给哪些专家
- All-to-All 通信将 token 发送到对应的 EP rank
- 计算完成后 All-to-All 将结果收集回来
```

##### 3.5.3.2 权重维度详解

##### 3.5.3.3 Routed Expert 计算流程

```python
# 假设当前 EP rank 负责专家 0 和专家 1
# 通过 All-to-All 接收分配给这些专家的 tokens

received_tokens = all_to_all_recv(...)  # [num_assigned_tokens, 7168]
received_expert_indices = ...  # 每个 token 应该由哪个专家处理

# 为专家 0 的 tokens 计算
mask_expert_0 = (received_expert_indices == 0)
tokens_expert_0 = received_tokens[mask_expert_0]  # [n0, 7168]

# 从 weight1 提取专家 0 的权重
gate_up_0 = experts.weight1[:, 0:24576]  # [7168, 24576]
down_0 = experts.weight2[0:12288, :]     # [12288, 7168]

# 专家 0 的前向计算
fc1_out_0 = tokens_expert_0 @ gate_up_0  # [n0, 24576]
gate_0, up_0 = fc1_out_0.chunk(2, dim=-1)  # 各 [n0, 12288]
activated_0 = F.silu(gate_0) * up_0      # [n0, 12288]
output_0 = activated_0 @ down_0          # [n0, 7168]

# 同样为专家 1 计算...

# 合并两个专家的输出
expert_outputs = concat([output_0, output_1])  # [num_assigned_tokens, 7168]

# All-to-All 将结果发送回原始位置
final_output = all_to_all_send(expert_outputs)
```

##### 3.5.3.4 Grouped GEMM 优化

实际实现中，为了效率会使用 **Grouped GEMM** 一次性处理多个专家：

```python
# 不使用 Grouped GEMM: 每个专家单独计算，低效
for expert_id in [0, 1]:
    tokens = get_tokens_for_expert(expert_id)
    output = expert_forward(tokens, expert_id)

# 使用 Grouped GEMM: 合并计算，高效
# 一次性对所有分配给当前 rank 的 tokens 计算
# 内部自动处理不同专家的权重切换
output = grouped_gemm(received_tokens, experts.weight1, expert_indices)
```

#### 3.5.4 MoE 训练优化技术

根据预训练脚本，Kimi2 使用了多种 MoE 优化技术：

##### 3.5.4.1 Grouped GEMM

```bash
--moe-grouped-gemm
```

**作用**：使用 Grouped GEMM 一次性计算多个专家的输出

```python
# 不使用 Grouped GEMM: 串行计算每个专家
for expert_id in selected_experts:
    output = expert_forward(tokens_for_expert[expert_id])

# 使用 Grouped GEMM: 合并计算
# 将所有分配给当前 rank 的 tokens 一起计算
output = grouped_gemm(all_tokens, all_expert_weights, expert_indices)
```

**优势**：

- 更大的矩阵乘法，提高 GPU 利用率
- 减少 kernel launch 开销
- 更好的内存访问模式

##### 3.5.4.2 Shared Expert Overlap

```bash
--moe-shared-expert-overlap
```

**作用**：将 Shared Expert 的计算与 Routed Experts 的计算重叠

```
标准执行:
┌──────────┬───────────────┬──────────┐
│  Shared  │    Routed     │  Merge   │
│  Expert  │   Experts     │  Output  │
└──────────┴───────────────┴──────────┘
   时间 →

Overlap 执行:
┌──────────┐
│  Shared  │
│  Expert  │
│     ┌────┴───────────┐
│     │    Routed      │
│     │   Experts      │
│     └────┬───────────┘
│          │  Merge     │
└──────────┴────────────┘
   时间 →
```

**优势**：减少总计算时间，提高硬件利用率

##### 3.5.4.3 MoE Zero Memory

```bash
--moe-zero-memory level0
```

**作用**：优化 MoE 层的内存使用

- **level0**: 基础优化，减少 activation 缓存
- **level1/2**: 更激进的优化，可能影响数值稳定性

##### 3.5.4.4 Token Permute/Unpermute 优化

```bash
--use-fused-moe-token-permute-and-unpermute
--moe-permutation-async-comm
```

**作用**：

- 使用融合的 kernel 进行 token 重排
- 异步通信与计算重叠
- 减少 All-to-All 通信的延迟

##### 3.5.4.5 DualPipeV 调度

```bash
--schedules-method dualpipev
--moe-fb-overlap
```

**DualPipeV**：DeepSeek 提出的流水线并行调度算法

```
传统 Pipeline Parallel:
- 每个 stage 顺序执行 forward/backward
- 气泡（bubble）时间较长

DualPipeV:
- 交错执行不同 micro-batch 的 forward/backward
- 减少流水线气泡
- 支持 MoE 前向/后向重叠
```

**优势**：

- 更高的流水线并行效率
- 特别适合 MoE 模型（有大量通信）

##### 3.5.4.6 激活重计算 (Activation Checkpointing)

```bash
--recompute-granularity full
--recompute-method block
--recompute-num-layers 1
--recompute-activation-function
```

**作用**：减少激活值内存占用

```
标准训练:
Forward → 保存所有激活 → Backward 使用保存的激活
内存占用: O(num_layers × seq_len × hidden_size)

激活重计算:
Forward → 只保存关键激活 → Backward 时重新计算部分激活
内存占用: O(√num_layers × seq_len × hidden_size)
```

**trade-off**：

- 内存 ↓↓ (大幅减少)
- 计算量 ↑ (约 20-30% 额外计算)
- 整体吞吐量 ↑ (可以增大 batch size)

> **注意**：MoE 专家权重（weight1、weight2）在 TP=1 和 TP=2 时**保持不变**，因为 EP 和 TP 是正交的并行策略。改变 TP 只会影响 Shared Expert 和 Router 的维度，不影响 Routed Experts 的存储方式。

## 总结

### 核心发现

1. **配置差异**
  - 本文档基于 `model_param_mapping_extracted.json`（4层模型，TP=1, PP=1, EP=1 dump）
  - 实际训练使用 `pretrain_kimi2_1t_4k.sh`（32层，TP=2, PP=8, EP=64）
  - EP=1 dump 包含**全部128个专家**的权重，EP=64 每 rank 只存 2 个专家
2. **模型架构特点**
  - **32层**：Layer 0-1 Dense MLP + Layer 2-31 MoE
  - **128专家**：每个 token 激活 2 个专家（1.56% 稀疏度）
  - **GQA + MLA**：64 头 Query，2 组 KV，支持 128K 上下文
  - **1T 参数**：稀疏激活，推理时约 3.9B 激活参数
3. **并行策略**
  - **TP (Tensor Parallel)**：分割 hidden 维度，影响 Attention/FFN
  - **EP (Expert Parallel)**：分割专家数量，每 rank 存 2 个专家
  - **TP 与 EP 正交**：可以组合使用（如 TP=2, EP=64）
4. **关键优化技术**
  - **Group-Limited Routing**：8 组，每组选 2 专家
  - **Grouped GEMM**：高效计算多专家
  - **DualPipeV**：流水线并行调度
  - **Activation Checkpointing**：减少激活内存
  - **Yarn RoPE**：32 倍长度扩展（4K→128K）
  - **SWA**：滑动窗口 Attention，降低长序列计算
5. **维度规律**
  - 词表/Attention/FFN 维度受 TP 影响
  - MoE 专家权重维度由 EP 决定，与 TP 无关
  - SwiGLU 激活：FFN 维度 × 2（gate + up）
